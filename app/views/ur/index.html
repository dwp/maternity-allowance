Round 1: 20 October 2021

Prototype iteration 1

Round 2: 26, 27 October 2021

Prototype iteration 2

  • We added details of the Highest earning weeks as some agents stated they needed to know what figures had been used.
  • We put the Highest 13 weeks inside a details toggle, because not all agents stated that they needed to see the full details, and adding the full list to the summary page made it cluttered and difficult to navigate.
' } }, { heading: { text: "Hypotheses" }, content: { html: '
  • We believe that agents who need to see the details of the highest earning weeks will find the information they need, and understand that the highest earning weeks have been selected for use in the calculation.
  • We believe agents will feel more confidence in the calculation when they can see the weekly earnings used.
  • We will know if this is true by observing agents in user research sessions and discussing their understanding and level of trust in the information displayed.
' } }, { heading: { text: "Research notes" }, content: { html: "

Number of participants:

" }, content: { html: '
  • Number of participants:
  • There was confusion amongst agents as to who the intended audience of the service was. Some thought that it was for citizens and still believed this upon completion.
  • Some agents entered a title into the full name field, this did not happen on the previous round so this is an area to observe moving forwards.
  • As with the previous round, agents weren’t sure where this information was from, there was slightly less recognition that it could be form a source such as RTI.
  • The highest earning weeks was noticed by some agents without the need for prompting. The information contained was reassuring to agents as it is similar to how the information is entered and displayed in legacy. Round 3: 4 November 2021

Prototype iteration 3

{% from "govuk/components/accordion/macro.njk" import govukAccordion %} {% from "govuk/components/accordion/macro.njk" import govukAccordion %} {{ govukAccordion({ id: "accordion-default", items: [ { heading: { text: "Design decisions" }, content: { html: '
  • We updated the labels on question fields to adhere to GDS patterns and guidance for pages with multiple questions.
  • We added a question to capture the date the claim was received into DWP, as this can impact the MAP.
  • We used radio options for Today, Yesterday, and Another date as we believe most claims would be processed within 2 days of receipt.
  • This can save agents inputting the same dates over and over again as they work through multiple cases each day.
  • We added a question to collect the claimant’s chosen MAP start date. We believe it is possible to make calculate whether the claimant is entitled to choose their MAP start date in the background, and display or not display the question dynamically, depending on the results of that calculation.
  • We added a banner to the summary page which is very similar to the Searchlight banner, as we believe this will help the agents feel more trust in the information displayed.
' } }, { heading: { text: "Hypotheses" }, content: { html: '
  • We believe agents will feel more trust in the information displayed when the screen looks like something they are used to interacting with.
  • **FUTURE SCENARIO TO TEST: We believe that agents will understand why the claimant’s chosen date and the actual MAP start date differ, if they are not the same.
  • We believe the agents will notice the difference between the MAP start dates without a visual cue on the summary page.
  • We’ll know this is true through observation of users in user research sessions.
' } }, { heading: { text: "Research notes" }, content: { html: "

Number of participants: 5

" }, content: { html: '
  • Number of participants: 5
  • The introduction of the MAP was received positively. Agents believed that this would be calculated based on some of the information we had asked them to enter during the data capture. The type of MAP rule applied could potentially be a missing feature.
  • The introduction of the MAP did also seem to create a need to know the 14th and 20th weeks for eligibility purposes.
  • Agents didn’t notice the Key Details bar.
  • Date claim received didn’t cause any issues, but it was highlighted that it’s very unlikely that agents would be working on claims from today or yesterday, so they would always need to enter a different date.
  • Employer name was highlighted as being needed. This has been heard in previous rounds of research. Currently this cannot be pulled through by the Finances team.
  • Some future areas to consider: alternative version, date of claim, non-flexi MAP scenarios.
' } } ] }) }}

  • To break up the long page of claim summary information, we decided to use tabs to break the content into more manageable sections.
  • Tabs are an experimental pattern in the Design System, and require further research.
  • To further our understanding of the MAP rules and the information agents require for cases which fall into each rule group, we created 2 scenarios for testing in this round of research.
  • Scenario 1 – The claimant’s baby has been born between the 11th and 4th week. The claimant’s chosen MAP start date is too far in the future. The MAP starts the day after the baby is born.
  • Scenario 2 – The claimant’s employment ended before the 11th week. Their MAP starts on the 11th week.
' } }, { heading: { text: "Hypotheses" }, content: { html: '
  • We believe that grouping the information into manageable sections, displayed in tabs, will prevent the user from being overwhelmed.
  • We believe that grouping the information into manageable sections, displayed in tabs, will enable the agent to easily digest the information displayed.
  • We believe that the tab headings, which are all displayed above the fold, will reassure agents that all the information they need to see is available. They will not need to scroll to check that it is all there.
  • We will observe user behaviour through usability testing to see if this is the case.

  • We believe the presence of the MAP rule on the decision tab will reassure agents that the claim has been worked out accurately.
